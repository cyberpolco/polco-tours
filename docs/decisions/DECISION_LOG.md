# POLCO TOURS — Decision & Change Log

Mirror of the governance record in the master design package. **Living-document
mandate (DR-007):** every structural change (data model, module boundary,
permission, business rule) and every integration change (add/remove/reconfigure
an external service) is recorded here with a dated entry, the affected volume(s)
updated, and the DR id referenced in the PR — before merge. Enforced by the PR
template and the Definition of Done (Vol. 10 §10.3).

| DR-014 | 2026-07-09 | First authenticated frontend: staff-only pilot dashboard (`src/app/staff/...`) for Lam to manage bookings/invoices/payments, sequenced ahead of a future tourist-facing site. Mounted Better Auth's own HTTP route for the first time (`src/app/api/auth/[...all]/route.ts`, `toNextJsHandler`) -- previously configured but never wired up. New `authService.getUserByEmail` (thin wrapper, no new DB capability) so staff can book on behalf of an existing tourist found by email. Business rule: staff may only book for tourists who already have an account this increment -- creating an account for a brand-new, never-signed-up client ("lead" capture) is explicitly deferred, since `Booking.touristUserId` is a hard FK and no such capability exists yet. Real payment collection during the pilot is still manual (`payment.resolve`) -- the DPO gateway remains stubbed (OI-01 still open); a "payment link" in the UI is real, but doesn't yet move real money. New CI job runs Playwright e2e (previously installed but never invoked in CI) against its own Postgres bootstrap, mirroring the `quality` job's setup. | V4, V5, V7, V9 |
| DR-015 | 2026-07-09 | Staff booking-setup wizard: extends the DR-014 "new booking" staff flow into a multi-step wizard (still staff-only; a tourist self-serve equivalent is a separate future increment). New tenant-scoped tables: `Traveler` (one per seat; exactly one `isTourLead=true` per booking regardless of group size; captures name/age/sex/nationality/ID-or-passport-number/phone/disabilities/allergies/drinkPreference), `Document` (generic passport-metadata store, reusable by Phase 2 visa documents -- `kind` is a plain string, not an enum, so adding `VISA` later needs no migration), `AddonService` (new org-scoped catalog concept in the `catalog` module -- priced photography/videography/translator/visa-assistance), `BookingAddon` (selection, price/currency snapshotted at selection time, same pattern as `Booking.priceMinor`). New `Booking.addonsFinalizedAt`, stamped by the add-ons wizard step (including choosing none) -- gates `bookingService.getBillableTotal`, the new cross-module entry point `invoicing` now calls instead of reading `Booking.priceMinor` directly, so an invoice's subtotal always includes finalized add-ons and can never be created before the traveler manifest + tour-lead passport + add-ons steps are all complete (a behavior change to `GET /bookings/{bookingId}/invoice`, now 409 until then). First real exercise of DR-010's object-storage decision: new `documents` module wraps `@vercel/blob` (`access: 'private'` -- no public URL ever exists for a passport; retrieval uses the SDK's authenticated `get()`, not a bare fetch) behind a `BlobGateway` interface (charter rule 8); passport bytes are only ever served back through an authenticated, audited `GET /bookings/{bookingId}/travelers/{travelerId}/passport` route -- the Blob pathname itself never reaches a client. RBAC change: `TOUR_OPERATOR` gains `documents.write` (previously read-only) so staff can upload a passport on a tourist's behalf. New static `src/lib/country-codes.ts` (no new dependency, charter rule 4) powers the traveler form's phone-country-code/flag and nationality pickers. `docs/design-package/` still doesn't exist in-repo to actually update -- flagged again, not fabricated. See OI-08 below: this repo's CI/e2e environments have no `BLOB_READ_WRITE_TOKEN`, so real end-to-end passport upload is only verified with the Blob gateway mocked at the test boundary (`tests/api/booking-setup.api.test.ts`); the Playwright e2e coverage deliberately stops at the passport step's upload form rather than submitting a real file. | V1, V4, V6, V7, V8 |
